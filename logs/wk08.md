## What I learned

### External APIs change without warning

REST Countries v3.1, which my original MP3 plan was built around, was deprecated mid-build. The replacement (v5) requires an API key on every request, which doesn't work for a static GitHub Pages site with no backend to hide a key behind. I had to find and switch to a different free, no-key, CORS-enabled alternative (countries.dev) without changing the rest of the app's architecture. Lesson: even a "no-key, stable" API pick can shift under you, and the fix is to check the _current_ official docs, not trust older tutorials or my own training data.

### Browser caching can hide a fix that actually worked

After fixing a bug in `script.js`, the live site kept showing the old broken behavior. The browser (and possibly GitHub Pages' CDN) was serving a stale cached copy of the JS file. Fixed by adding a cache-busting version query string (`script.js?v=2`) to the script tag, and learned to always hard-refresh (Cmd+Shift+R) and check the Network tab before assuming a fix didn't work.

### Lighthouse catches real bugs, not just style nitpicks

- `loading="lazy"` on images that are visible immediately on page load actually _hurts_ performance (delays LCP), it's not just a "nice to have."
- `role="button"` on a semantic element like `<article>` is technically ARIA-invalid; the fix was switching to a plain `<div>` with the role.
- Lighthouse caught that one country's flag image was loading from an oddly-labeled Wikipedia asset, which also triggered a third-party cookie warning. Fixed by sourcing all flags from one deterministic CDN (flagcdn.com) instead of trusting whatever URL the country API happened to return.

### Code organization: splitting data from logic

Using `export`/`import` to keep a data array in one file (`data.js`) and rendering logic in another (`main.js`) means the data source can change later (e.g. to a real API or database) without touching the code that displays it. ES modules require a real server to test though, `file://` won't work, only `http://` via Live Server.

## AI interactions

### Interaction 1 (the required specific one)

- **What I asked**: To show live currency conversion to INR in the MP3 detail view, using the Frankfurter API.
- **What AI gave me**: Code that fetched exchange rates with INR as the base currency and displayed the result as `"1 INR = 0.012 USD"`, which is mathematically correct.
- **What I kept / changed / rejected**: I flagged that this read backwards for actual use, nobody comparing prices wants to see "how much foreign currency is 1 rupee," they want "how much does this cost in rupees." I asked for it to be inverted, and the code was changed to compute `1 / rate` and display `"1 USD = ₹83.50"` instead, using the ₹ symbol directly. **How I knew it needed the change**: the direction only became obvious once I looked at the actual detail panel and tried to use it the way a real Realply employee would, checking what a foreign price costs in rupees. The math wasn't wrong, the direction just wasn't useful.

### Interaction 2

- **What I asked**: Why non-INR currency conversions were still showing "not available" after the direction fix above.
- **What AI gave me**: A hypothesis that it was a browser caching issue, plus a debugging path (Network tab, check the actual request URL).
- **What I kept / changed / rejected**: The hypothesis was right in spirit but incomplete, checking the Network tab revealed the real cause was a second issue: Frankfurter's actual current endpoint uses a different domain and parameter (`api.frankfurter.dev` with `base=`) than what was originally coded (`api.frankfurter.app` with `from=`), the same kind of silent API deprecation as the REST Countries issue. I kept the caching fix (version-busting the script tag) since that was also a real, separate problem, but the root cause needed a second, different fix.

### Interaction 3

- **What I asked**: Built `playlist.html` by hand first (array of songs, a `for...of` loop rendering one `<li>` per song), then asked AI to extend it with `push` and `unshift` controls so I could see the difference between adding to the end vs. the start of the array.
- **What AI gave me**: Two buttons wired to `songs.push(value)` and `songs.unshift(value)`, both calling the same `renderPlaylist()` function afterward to re-render from the updated array.
- **What I kept / changed / rejected**: Kept it as-is, it's a direct, correct implementation of the render-from-data pattern the exercise was testing.

### Interaction 4

- **What I asked**: Built the core of `fetch-fun.html` by hand (one `async` function fetching a random dog image, reading `data.message`, wrapped in `try`/`catch`), then asked AI to add a "Surprise me" button that randomly picks between two different APIs.
- **What AI gave me**: A second fetch function for a different API (Advice Slip) reading a differently-shaped response (`data.slip.advice` instead of `data.message`), plus a button using `Math.random() < 0.5` to pick between them.
- **What I kept / changed / rejected**: Kept it. Verified both API endpoints were actually still live and CORS-enabled before trusting the code, given I'd already been burned twice this week by APIs changing underneath a working build (see REST Countries and Frankfurter above).
