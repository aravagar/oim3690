## What I learned

### Iteration 1: HTML structure and first interaction

For MP2 I chose a sidebar layout with the Add Order form on the left and the order board on the right. The semantic elements used were `<header>`, `<aside>` for the sidebar, `<main>` for the board, `<article>` for each order card, and `<fieldset>` with `<legend>` for the form sections. This felt right because the form and the board are two distinct areas with different jobs.

The first piece of JavaScript was the form submit event listener. It reads the input values, builds a new order object, pushes it into the `orders` array, and calls `renderOrders()` to redraw the board. What I had to understand to make it work was why `renderOrders()` clears and rebuilds the entire list from the array rather than trying to update one card at a time. The reason is that rebuilding from state keeps the DOM and the data in sync automatically on every change.

### Iteration 2: Core functionality

The hardest interaction to implement was the inline reset confirmation. The assignment says no `alert()` or `confirm()` boxes, so I had to build a two-step click pattern entirely in the DOM: first click changes the button text and injects a warning box, second click executes the reset. This required a `pendingResetConfirm` boolean to track which click the user was on.

JavaScript concepts used across the build:
- **Arrays and array methods**: `filter`, `map`, `find`, `push` to manage the `orders` array
- **Functions**: separate functions for rendering, building cards, updating the dashboard, advancing stages
- **Conditionals**: `if/else` for validation, overdue detection, edit mode gating, filter logic
- **DOM methods**: `createElement`, `appendChild`, `classList.toggle`, `querySelector`, `dataset`
- **Event listeners**: form submit, button click, dropdown change, keyboard input, backdrop click

Bugs found and fixed:
- A blank SKU row was being added before edit mode was unlocked, so it appeared invisibly. Fixed by only calling `addSkuRow()` when edit mode is entered, not on init.
- Advance stage buttons were not re-rendering correctly after exiting and re-entering edit mode. Fixed by calling `renderOrders()` on both `enterEditMode()` and `exitEditMode()`.

### Iteration 3: Polish and sub-stages

Feedback from testing: the app needed to reflect Studioforma's real production workflow more accurately. Added Carpentry, Painting, and Finishing as sub-stages inside "In Production", displayed as a segmented progress track on each order card. The "Move to Quality Check" button is locked until the order reaches Finishing, which prevents skipping steps.

Edge cases handled:
- Rapid clicking on advance buttons (lock Set that ignores clicks within 600ms)
- Removing the last SKU row (blocked with an inline error message)
- Filter showing zero results (clear message instead of blank space)
- localStorage corruption on load (try/catch falls back to an empty board)

With more time I would add: the ability to edit an order after it is created, export to CSV so Mr. Laxman can share a report, and real shared persistence via an API so the whole team sees the same board across devices.

### Data types and functions (6/24 class)

The key lesson from the converter exercise: form inputs always arrive as strings, even if they look like numbers. `'5' + '3'` gives `'53'` instead of `8` because JavaScript joins strings instead of adding them. Wrapping the input in `parseFloat()` before doing math fixes this. Used `isNaN()` to check whether the input is actually a number before calculating, and `.toFixed(2)` to round the result to two decimal places.

The difference between `return` and `console.log`: `console.log` only prints to the console for debugging, the function still returns `undefined`. `return` hands the value back so the rest of the code can use it. The `getDiscount` function in `discount.html` uses `return` so the checkout page can take the result and calculate the final total.
