## What I learned

### Isolating the data layer is worth the extra file

Every localStorage read and write in the app happens in `store.js` and nowhere else. `app.js` never sees `localStorage` at all, it just calls `loadData()`, `createProject()`, `advanceProject()` and so on.

The payoff is that swapping to a real database later means rewriting one file instead of hunting through 970 lines of view code for storage calls. This was a deliberate choice from the tech stack class on 7/13, and it is the single decision I would defend hardest in this project.

### Changing your data model breaks saved data, and you have to plan for it

I changed the shape of a project twice mid-build: first replacing a single `productNeeded` field with a list of items, then splitting a flat `value` into `qty` and `unitPrice`. Anyone who had already saved data would have had it break on load.

```js
function migrate(data) {
  data.projects.forEach(p => {
    if (!Array.isArray(p.items)) {
      p.items = [];
      // fold the old single-product fields into the new list
    }
  });
}
```

It runs once on load and converts old records forward. Nothing I have written before this needed one, because nothing before this had users with data already in it.

### Put the human at the last step, on purpose

The status summary does **not** send itself. It opens WhatsApp with the message already written and a person presses send.

That was deliberate. A wrong status message to an architect who is quoting us to their client is expensive, and the check costs two seconds. Dr. Li picked this out in the wrap-up as a general principle: decide where the human step goes before you build it, not after something embarrassing goes out.

Also a small practical thing: `wa.me` needs the number in international format with no `+`, no spaces and no dashes, so `919XXXXXXXXX`. I wrote a `normalizePhone()` helper because nobody types a number that way.

### A browser extension is just a folder

From the 7/20 class. HTML, CSS and JS in a folder, plus one file called `manifest.json` that tells Chrome what it is. Load unpacked from `chrome://extensions` with developer mode on and it runs. The one thing to check when AI writes extension code is `"manifest_version": 3`, because V2 answers still float around and no longer load.

### Commit before you let an agent run

`git add -A && git commit` before handing a folder to a coding agent. Then `git diff` shows exactly what it changed and `git checkout .` undoes all of it in one command. I did not do this consistently early in the week and lost work I had to redo.

### Green tests do not mean correct behaviour

The class demo had an agent write its own tests, pass all of them, and still count HTML comments as words. It tested what it decided to test. The habit I took from it and now use every time:

> When you finish, list every decision you made that I did not specify.

That prompt has caught more real problems for me than reading the diff has.

### The hard part was not the code

The code came out fast. What took real time was deciding what the stage names should actually be, whether the delivery date should go to a client by default (no), and whether Move Back should demand a reason (yes, otherwise the history log is useless). Those are product decisions, and no tool made them for me.

## AI interactions

### Interaction 1

- **What I asked**: To write `PRD.md` for the CRM before writing any code, covering what it is for, who uses it, and what version one does.
- **What AI gave me**: A full requirements document with numbered functional requirements (FR1 through FR8) and a technical considerations section.
- **What I kept / changed / rejected**: Kept it, and then actually used it, which is the part that mattered. I referenced the FR numbers in code comments as I built, so `renameStage()` says "PRD: FR5" above it. Reading one page carefully and catching a wrong requirement there is far cheaper than catching it after 900 lines are written on top of it.

### Interaction 2

- **What I asked**: My previously saved projects broke after I changed a project from having one product to having a list of items. I asked how to handle it.
- **What AI gave me**: A `migrate()` function that detects the old shape on load, converts it forward, and saves the converted version back.
- **What I kept / changed / rejected**: Kept the pattern, changed the scope. It wrote a migration for one change. I extended it to handle both model changes and had it record a `version` number in the stored object, so the next change has somewhere to hook into. This was a concept I did not know existed at the start of the week.

### Interaction 3

- **What I asked**: To wire up the WhatsApp status message.
- **What AI gave me**: A working `wa.me` link, and it proposed firing the message automatically when a project advanced a stage.
- **What I kept / changed / rejected**: **Rejected the automatic send.** Kept the link, kept the phone normalization, and made it a button the user presses with the message pre-filled. I also made the internal delivery date and full stage timeline opt-in checkboxes rather than always included, because a client does not need to see that a project sat in Quote Shared for three weeks. The default should be the safe one.
