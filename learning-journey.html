04

What the failures taught

Four things I only learned by getting them wrong.

01

An API you depend on can die while you are building on it

REST Countries went dead mid-build and Frankfurter moved without telling anyone. Now I check that a service is current, keyless and CORS-enabled before I write a line against it.

02

The browser will happily show you a bug you already fixed

Stale cached JavaScript cost me an afternoon of debugging working code. app.js?v=7 and a hard refresh, then believe what the Network tab says over what the page says.

03

Correct and useful are different things

"1 INR = 0.012 USD" is not wrong. It is just backwards for anyone at Realply checking what a foreign price costs. I only saw it by using the panel the way an employee would.

04

Passing tests written by the same tool prove very little

An agent tests what it decided to test. So I ask it directly: list every decision you made that I did not specify. That question has found more real problems than reading the diff.

05

Still open

What I am carrying out of the course.

Every log ended with a questions section. These are the ones the semester did not close, and they are the reason this does not stop at the last checkpoint.

  • Is store.js really a drop-in swap to Supabase because the data layer is isolated, or are auth and sync the actual job hiding behind that claim?
  • Once several people write to the same board, what happens when two of them advance the same project at the same time?
  • How do you tell in advance whether an API key is safe to expose client-side, when most providers never say either way?
  • To make the CRM something the team opens daily instead of a prototype, is the blocker the shared backend or the experience on a phone?
